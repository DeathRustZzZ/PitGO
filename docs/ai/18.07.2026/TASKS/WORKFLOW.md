# WORKFLOW — как работать с Claude Code над задачами PitGO

Цель процесса: **код пишешь ты**, Claude — архитектор-наставник и ревьюер. Claude объясняет, подсказывает и проверяет, но не реализует задачи за тебя (кроме случаев, когда ты явно попросишь показать фрагмент).

## Цикл работы над задачей

1. **Выбери задачу** — первую неотмеченную в [ROADMAP.md](ROADMAP.md), открой её файл в [TASKS/](TASKS/). Задачи упорядочены по зависимостям — не перепрыгивай.
2. **Попроси объяснение без реализации.** Пример промпта:
   > Прочитай docs/ai/TASKS/001-ownership-invariant.md. Объясни задачу: какие файлы и почему затронуты, какие решения мне предстоит принять, с чего начать. Код не пиши.
3. **Реализуй задачу сам.** Держись Scope (что можно/нельзя менять) и Step-by-step plan. Застрял — проси подсказку, а не решение:
   > Я делаю шаг 4 задачи 001. Не понимаю, как X. Дай направление/подскажи идиому, но не пиши реализацию за меня.
   Хорошие формы вопросов: «какая идиома тут уместна», «почему компилятор ругается на …», «какие есть варианты и их trade-offs».
4. **Прогони проверки** (то же делает pre-commit hook):
   ```bash
   cargo fmt --all --check
   cargo clippy --workspace --all-targets --all-features -- -D warnings
   cargo test --workspace
   ```
   (Для задач, трогающих frontend, — ещё `npm --prefix frontend run check`.)
5. **Попроси ревью diff:**
   > Проведи ревью моего diff по docs/ai/REVIEW_CHECKLIST.md в контексте задачи docs/ai/TASKS/00X-….md. Сначала запусти git diff. Формат: Must fix / Should fix / Nice to have / вердикт.
6. **Claude отвечает строго в формате:**
   - **Must fix** — блокирует: баги, нарушение границ слоёв, `unwrap` в проде, непокрытое поведение;
   - **Should fix** — стоит исправить сейчас;
   - **Nice to have** — идеи на потом (кандидаты в ROADMAP);
   - **Final verdict: approve / request changes** — с одним предложением-обоснованием.
7. **Исправь замечания** (минимум — все Must fix) и повтори шаги 4–5 до approve.
8. **Закоммить сам.** Сообщение — из раздела «Suggested commit message» задачи (conventional commits: `type(scope): описание`). Один коммит на задачу — норма; логичные промежуточные коммиты тоже допустимы.
9. **Перейди к следующей задаче:** отметь чекбокс в ROADMAP.md, вернись к шагу 1.

## Правила для Claude в этом проекте

- Не реализовывать задачи из TASKS/ — только объяснять, подсказывать, ревьюить.
- В ревью опираться на REVIEW_CHECKLIST.md и Scope конкретной задачи; несвязанные изменения — всегда как минимум Should fix.
- Подсказки давать по возрастанию конкретности: идея → направление → идиома → (только по явной просьбе) фрагмент кода.
- Не предлагать расширение объёма задачи; идеи фиксировать как Nice to have.

## Правила для тебя

- Одна задача за раз; не смешивай две задачи в одном diff/коммите.
- Если по ходу нашёл чужой баг или хочешь рефакторинг — запиши (попроси Claude добавить в ROADMAP), но не делай сейчас.
- Если задача оказалась крупнее, чем выглядела, — останавливайся и проси Claude разбить её, а не расползайся по Scope.
- Вопросы «почему так спроектировано» — задавай: half of the value этого проекта — в обсуждениях, а не в коде.

## Когда фаза закончена

Все 5 чекбоксов фазы отмечены → попроси Claude:
> Фаза N завершена. Проведи короткое повторное ревью затронутой области, обнови docs/ai/PROJECT_REVIEW.md (раздел «прогресс») и детализируй задачи следующей фазы в docs/ai/TASKS/ (следующие 5 файлов по тому же шаблону).
