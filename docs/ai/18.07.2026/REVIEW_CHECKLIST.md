# REVIEW_CHECKLIST — PitGO

Общий чек-лист, по которому Claude проводит ревью после каждой выполненной задачи. Ревью идёт по `git diff` относительно последнего коммита. Каждая находка получает категорию **Must fix** / **Should fix** / **Nice to have**; итог — **approve** или **request changes**.

## 1. Корректность

- [ ] Решает ли diff именно задачу из task-файла (Goal), ничего не упуская и не добавляя лишнего?
- [ ] Все ветки поведения обработаны (Some/None, Ok/Err, каждый вариант enum)? Exhaustive match предпочтён «catch-all `_`», особенно на доменных enum.
- [ ] Идемпотентность/no-op семантика не сломана (повтор команды по-прежнему даёт `NoChange`, где положено)?
- [ ] Версии агрегатов: инкремент ровно на 1 на событие; optimistic-locking-проверки не обойдены.
- [ ] Нет незамеченных изменений публичного контракта (маршруты, статусы, коды и формы JSON) — если контракт меняется, это явно заявлено задачей.

## 2. Архитектурные границы

- [ ] Направление зависимостей: `domain → shared`; `application → domain`; `infrastructure → application+domain`; `backend` — composition root. Ни одного нового импорта против стрелок.
- [ ] Domain без serde/HTTP/SQL/инфраструктурных типов; бизнес-правила не переехали в репозитории или роутеры.
- [ ] Application не знает про axum/HTTP-статусы; backend не вызывает домен в обход application (маппинг доменных типов в DTO — допустимое исключение, конструирование/команды — нет).
- [ ] DTO и сериализация — только в backend; конвертация DTO↔domain явная.
- [ ] Порты (traits) меняются осознанно: изменение порта — изменение контракта, отражено в моках и реализациях.

## 3. Идиомы Rust

- [ ] Именование по конвенциям (snake_case, типы CamelCase); имена отражают домен, а не реализацию.
- [ ] Владение/заимствование без лишних `clone()` (кроме задокументированного техдолга `Clone` на агрегатах).
- [ ] `Option`/`Result` комбинируются идиоматично (`?`, `map`, `ok_or`) — но не в ущерб читаемости.
- [ ] Новые публичные API имеют doc-комментарии в стиле проекта; `#[must_use]` там, где игнорирование результата — баг.
- [ ] Нет `unsafe`, нет ненужных `pub` (минимальная видимость).

## 4. Обработка ошибок

- [ ] **Никаких `unwrap`/`expect`/`panic!` в production-коде** — только в тестах, либо с обоснованием в комментарии, почему инвариант делает панику недостижимой (например, bind при старте процесса).
- [ ] Ошибки типизированы (`thiserror`), несут достаточно контекста; сообщения не врут (doc-комментарий соответствует варианту).
- [ ] Разные причины — разные варианты: не переиспользуется чужой вариант «потому что похож».
- [ ] На HTTP-границе внутренние детали (тексты ошибок Mutex, версии, SQL) не утекают в `message`; machine-readable код стабилен и семантичен.
- [ ] Ошибки не проглатываются (`let _ =`, пустой `if let Err`) без явного обоснования.

## 5. Тесты

- [ ] Новое/изменённое поведение покрыто тестами; тесты падали бы без изменения кода (проверить мысленно: «что этот тест ловит?»).
- [ ] Тесты поведенческие: проверяют наблюдаемый результат (ошибку, состояние, ответ), а не внутреннее устройство.
- [ ] Один тест — одно утверждение о поведении; имя теста читается как спецификация.
- [ ] Ошибочные пути покрыты не хуже happy path.
- [ ] Тесты изолированы (нет разделяемого состояния, порядок не важен).
- [ ] Существующие тесты не ослаблены «чтобы прошло» (изменение ассертов — только если контракт изменён задачей).

## 6. Именование

- [ ] Имена методов говорят правду о семантике (`has_open_ownership` не считает только Active).
- [ ] Терминология единообразна во всех слоях и в трёх срезах (customer/vehicle/ownership не разъезжаются).
- [ ] DTO-поля соответствуют принятому стилю API (snake_case), доменные типы — доменному языку из docs/domain.

## 7. Простота

- [ ] Решение — минимально достаточное для Goal; diff маленький и сфокусированный.
- [ ] Нет «на всякий случай»: неиспользуемых параметров, преждевременных generic'ов, лишних слоёв косвенности.
- [ ] Копия из соседнего среза лучше преждевременного обобщения (три похожих репозитория — нормально; обобщать — когда появится четвёртый или боль).

## 8. Отсутствие оверинжиниринга

- [ ] Не добавлены новые крейты/зависимости без необходимости, заявленной задачей.
- [ ] Не введены трейты с единственной реализацией «для тестируемости», если тестируемость уже есть.
- [ ] Не создан фреймворк там, где хватало функции (особенно в тестовых хелперах).
- [ ] Абстракция вводится по правилу задачи (Scope/Hints), а не по вкусу.

## 9. Отсутствие несвязанных изменений

- [ ] В diff нет файлов вне Scope задачи; форматирование чужого кода не менялось.
- [ ] Нет попутных переименований/рефакторингов «раз уж я здесь» — такие идеи выписываются в заметку для будущих задач.
- [ ] Cargo.toml меняется только в объёме, разрешённом задачей; версии — через `[workspace.dependencies]`.

## Формат вердикта ревью

```
### Must fix        # блокирует approve: баги, нарушения границ, unwrap в проде, дыры тестов
### Should fix      # стоит исправить в этой задаче, но не блокирует при наличии причин
### Nice to have    # идеи на будущее; можно выписать в ROADMAP
### Вердикт: approve | request changes   # + одно предложение-обоснование
```
